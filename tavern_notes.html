1.2 TODO Motivation

Micmac is a very powerful 3D reconstruction and automatic point matching tool, but it's very complex to use for multiple reasons :

  • 3D reconstruction in itself is a complex process, that involves multiples steps tied together in sometimes complex ways that depend on the parameters of the scene, the target , the acquisition protocol and the quality of the pictures
  • Micmac tries to be a swiss-army knife, that provides a wide range of features without really specializing on a particular subset
  • Micmac has a complex interface that lacks error handling and that is not properly documented
  • Micmac does not provide an easy way to define automatic reconstruction profiles

Because of these reasons, we decided to create a "Micmac wrapper", that would simplify the usage patterns, implement proper error handling, fix Micmac's weird behaviors, and provide a base for additional features.

The name "Tavern" was chosen because of the names of Micmac's sub-commands.

1.2.1 TODO Secondary motivation : integration of the pipeline with Blender

Blender "Nodes"
Blender Manual > Composite Nodes

While developing Tavern, we noticed that the "pipelines" (see Pipelines) are similar to what is achieved using the Nodes of Blender.

couscous had the idea of using this to integrate Tavern with Blender, and to use nodes to define reconstruction pipelines, thus allowing to do the 3D reconstruction from a Blender-based interface.

This is also made possible by the fact that Tavern and Blender are both implemented using the same language.

1.3 TODO Targeted implementation

1.3.1 TODO Users

Tavern is primarily targeted at 3den, and was started because of a strong requirement for a sane way to use Micmac. However, we want it to be as generic as possible, and to be used wherever it can help.

1.3.2 TODO Use cases

We want Tavern to be able to handle the following situations :

Simple reconstruction (single set, calibration provided)
This is equivalent to what was being run by the reconstruction script in the Seafile
Simple calibration (single set, acquisition protocol defined)
When doing the initial calibration for a camera captor, we use a single image set made using a predefined protocol and a special target, and we want Tavern to use that to output calibration data that can be used later
Experimentations (many sets, complex work-flow)
When doing experiments on 3D reconstruction, we will have to define complex work-flows with multiple image sets and with multiple passes for some steps
Predefined profiles (adapted to acquisition protocol)
For example in the case of archaeological uses of Tavern, huge image sets made with a clearly defined acquisition protocol will be provided : in that case, predefined profiles can be adapted to the image sets and used to run the reconstruction

Later on, Tavern shall be extended to handle the following situations :

Blender-based reconstruction
Secondary motivation : integration of the pipeline with Blender
Web-based reconstruction
Pipeline defined in a web interface, reconstruction executed on the server. This is basically MaaS.

NOTE : The last feature, automatic reconstruction, could be extended by a "kind of" machine learning algorithm, that would try its best to guess the best settings for the 3D reconstruction. This would require some amount of computing power (a reconstruction farm), but the produced data could be very interesting and could make us learn a lot from Micmac.

Factories are always used in two distinct steps :

Initialization (__init__)
store the factory arguments
Execution (__call__)
execute the factory with the stored arguments

When a factory is called multiple times, it is only executed the first time.

Because of the way factories are structured (the they are used in two steps, the second one being idempotent), they can be assembled in pipelines in a very simple way.

When a factory depends on another factory, it should do two things :

  • Store the other factory during the initialization
  • Call the other factory during the execution
